# AI 网关压测落地方案

## 1. 压测目标

本次压测不仅关注 QPS 上限，还需验证以下目标：

1. 网关是否稳定可用
2. 用户是否感知响应迅速
3. 系统在目标并发下是否能够长期稳定运行
4. 上游模型异常时，网关是否具备容错能力

## 2. 压测对象拆分

建议将 AI 网关能力拆分为以下 6 类分别压测，避免混测导致结果失真。

### 2.1 核心推理转发链路

典型接口：

- `/v1/chat/completions`
- `/v1/completions`
- `/v1/responses`

这是最核心链路。

**思考推理模型特殊性**：
- 传统模型：请求 → 直接生成 → 响应
- 思考模型：请求 → 思考阶段 → 生成阶段 → 响应
- 指标差异：需要区分思考阶段和输出阶段性能

### 2.2 思考推理模型链路

针对支持推理的模型（如 OpenAI o1 系列、Claude 推理增强），需要专门测试：

**关键指标**：
- 思考时长（Thinking Duration）：请求开始到思考结束
- 输出 TTFT（Output TTFT）：思考结束后到第一个输出token
- 输出 TPS（Output TPS）：输出阶段的token速率
- 思考占比（Thinking Ratio）：思考时间占总时间的比例

**推理等级测试**：
- OFF：禁用思考，等同于传统模型
- LOW：轻度思考（2-5s）
- MEDIUM：中度思考（5-15s）
- HIGH：深度思考（15-60s）

### 2.3 场景化测试链路

**上下文长度维度**：
- 短上下文：< 2K tokens（快速问答）
- 中等上下文：2K-8K tokens（常规业务）
- 长上下文：> 8K tokens（文档分析）

**问题复杂度维度**：
- 简单问题：事实查询、简单计算
- 中等问题：推理分析、对比说明
- 复杂问题：多步推理、综合分析

**场景组合测试**：
- 短+简单：快速响应场景
- 中等+中等：常规业务场景
- 长+复杂：深度分析场景

### 2.4 流式输出链路

重点关注：

- 首 Token 时间（TTFT）
- 流式中断率
- 整体完成时间（TTLT）

### 2.5 非流式链路

重点关注：

- 总响应时间
- 错误率
- 吞吐能力

### 2.6 网关管理能力

例如：

- 鉴权
- 配额校验
- 模型路由
- 限流
- fallback

该类能力应单独压测，避免被模型推理时延掩盖。

## 3. 压测前必须确认的基线信息

正式压测前，需要补齐以下信息，否则测试结果缺乏解释力。

### 3.1 测试环境信息

- AI 网关部署节点数
- 单节点 CPU / 内存 / 带宽
- 网关实例副本数
- 上游模型类型与部署位置
- 是否走本地模型 / 云模型 / 混合路由
- 是否开启缓存、限流、重试、fallback

### 3.2 请求画像

至少需要明确：

- 平均输入 Token
- 平均输出 Token
- 流式占比
- 各模型调用占比
- **思考推理等级占比**（新增）
- **场景分布情况**（新增）
- 高峰并发预估
- 峰值 QPS 预估

缺少真实数据时，可先采用业务假设，例如：

- 60% 普通问答：输入 800 Tokens，输出 400 Tokens
- 20% 长文本总结：输入 6000 Tokens，输出 1200 Tokens
- 10% 思考推理：输入 1500 Tokens，输出 800 Tokens，思考时间 10-20s
- 5% 复杂分析：输入 5000 Tokens，输出 1500 Tokens，思考时间 30-60s
- 5% embeddings / rerank / 管理请求

**场景画像示例**：

| 场景类型 | 占比 | 输入Token | 输出Token | 推理等级 | 预期思考时长 | 预期总时长 |
| --- | --- | --- | --- | --- | --- | --- |
| 快速问答 | 40% | 200 | 150 | OFF | 0s | 1-2s |
| 简单推理 | 25% | 500 | 300 | LOW | 3-5s | 5-8s |
| 常规推理 | 20% | 1500 | 600 | MEDIUM | 10-15s | 15-20s |
| 深度分析 | 10% | 5000 | 1500 | HIGH | 30-45s | 40-60s |
| 文档处理 | 5% | 10000 | 2000 | MEDIUM | 20-30s | 30-40s |

## 4. 建议的压测指标

### 4.1 通用指标

- 请求总数
- 成功率
- 错误率
- 超时率
- 平均响应时间
- P95 / P99 响应时间
- QPS / RPS
- 并发数

### 4.2 AI 专项指标

**传统模型指标**：
- TTFT：首 Token 时间
- TTLT：最后 Token 完成时间
- Tokens/s：输出速率
- Token Throughput：每秒处理 Token 总量
- 流式中断率
- fallback 成功率
- 重试成功率

**思考推理模型指标**（新增）：
- **思考阶段指标**：
  - 思考时长（Thinking Duration）：从请求开始到思考结束
  - 思考 Token 数（Thinking Tokens）：思考过程的token数量（如API返回）
  - 思考时间占比（Thinking Ratio）：思考时间占总时间的比例

- **输出阶段指标**：
  - 输出 TTFT（Output TTFT）：思考结束到第一个输出token的时间
  - 输出 TPS（Output TPS）：输出阶段的token速率
  - 输出 TTLT（Output TTLT）：输出阶段的总时长

- **组合指标**：
  - 端到端延迟（End-to-End Latency）：完整的请求-响应时间
  - 场景化性能：按上下文长度和复杂度分组的性能表现

**场景维度指标**（新增）：
- **上下文长度影响**：
  - 短上下文平均思考时长
  - 中等上下文平均思考时长
  - 长上下文平均思考时长
  - 上下文长度影响系数

- **复杂度影响**：
  - 简单问题平均思考时长
  - 中等问题平均思考时长
  - 复杂问题平均思考时长
  - 复杂度影响系数

### 4.3 系统资源指标

- CPU
- 内存
- 网络带宽
- 连接池使用率
- 线程池活跃数
- JVM 堆 / GC（如果网关基于 Java）
- 容器重启次数
- Pod CPU throttle（如果运行于 K8s）

## 5. 压测场景设计

建议至少覆盖以下 6 类场景。

### 5.1 基线测试

目的：测单请求纯能力基线。

配置建议：

- 并发 1~5
- 持续 5~10 分钟
- 流式与非流式分别执行

观察项：

- TTFT 基线
- TTLT 基线
- 输出速度基线
- 单请求成功率

### 5.2 阶梯升压测试

目的：找到系统性能拐点。

配置建议：

- 并发按 `10 -> 20 -> 50 -> 100 -> 200 -> 300` 逐级提升
- 每级持续 10 分钟
- 每级间隔 2~3 分钟

观察项：

- 从哪一级开始 TTFT 明显抖动
- 从哪一级开始错误率上升
- 从哪一级开始 CPU / 线程池 / 连接池打满
- 从哪一级开始上游模型 429 / 5xx 增多

输出结论：

- 最大稳定并发
- 性能拐点

### 5.3 稳定性测试

目的：验证长时间运行不发生明显劣化。

配置建议：

- 目标并发取预估线上峰值并发的 70%~100%
- 持续 1 小时起步，建议 2 小时

观察项：

- P95 TTFT 是否持续升高
- 错误率是否持续爬升
- 内存是否持续上涨
- 连接是否泄漏
- 网关是否出现线程堆积

验收建议：

- 性能波动不超过 20%
- 无持续性错误积累
- 无实例异常重启

### 5.4 峰值突刺测试

目的：验证突发流量承载和保护能力。

配置建议：

- 从 50 并发瞬间拉升到 300 或 500
- 保持 3~5 分钟
- 再回落到正常值

观察项：

- 是否触发限流
- 是否出现请求堆积
- 是否发生雪崩
- 回落后是否可以恢复

重点不在于“全成功”，而在于：

- 是否可控失败
- 是否自动恢复
- 核心链路是否保住

### 5.5 混合流量测试

目的：贴近真实业务场景。

**传统混合比例**：
- 60% 流式 chat
- 20% 非流式长文本生成
- 10% embedding
- 10% 管理接口

**思考推理场景混合比例**：
- 40% 传统快速响应（OFF/LOW等级）
- 30% 常规推理（MEDIUM等级）
- 20% 深度分析（HIGH等级）
- 10% 长文档处理

**场景维度混合比例**：
- 50% 短上下文场景（< 2K tokens）
- 30% 中等上下文场景（2K-8K tokens）
- 20% 长上下文场景（> 8K tokens）

观察项：

- 不同接口之间是否互相影响
- 不同推理等级之间的资源竞争
- 不同场景之间的性能隔离效果
- 管理接口是否被推理流量拖慢
- 小请求是否被大请求拖垮
- 思考模式是否影响传统模式性能

### 5.6 异常容错测试

目的：验证网关容错价值。

建议模拟：

- 上游模型 429
- 上游模型 5xx
- 上游响应超时
- 某个模型实例不可用
- DNS / 网络抖动
- fallback 模型响应较慢
- **思考模式超时降级**（新增）
- **长上下文资源耗尽**（新增）

观察项：

- fallback 是否生效
- 重试是否合理
- 是否发生重试风暴
- 核心业务成功率是否可接受
- **思考模式超时后的降级效果**（新增）
- **不同场景之间的故障隔离**（新增）

## 7. 验收口径

建议将以下标准作为第一轮验收基线。

### 7.1 流式推理接口（传统模式）

- 成功率 >= 99.9%
- 错误率 <= 0.1%
- 超时率 <= 0.5%
- P95 TTFT <= 2s
- P99 TTFT <= 3s
- 平均输出速率 >= 15 tokens/s

### 7.2 思考推理接口验收标准（新增）

#### 7.2.1 按推理等级验收

| 推理等级 | P95 思考时长 | P99 思考时长 | 输出 TPS | 端到端 P95 | 适用场景 |
| --- | --- | --- | --- | --- | --- |
| OFF | N/A | N/A | ≥20 | ≤2s | 快速响应场景 |
| LOW | ≤5s | ≤8s | ≥20 | ≤8s | 简单推理场景 |
| MEDIUM | ≤20s | ≤30s | ≥20 | ≤25s | 复杂推理场景 |
| HIGH | ≤45s | ≤60s | ≥15 | ≤65s | 深度分析场景 |

#### 7.2.2 按场景组合验收

| 场景类型 | P95 思考时长 | 输出 TPS | 成功率 | 业务价值 |
| --- | --- | --- | --- | --- |
| 短+简单+LOW | ≤5s | ≥20 | ≥99.9% | 快速响应 |
| 中等+中等+MEDIUM | ≤20s | ≥20 | ≥99.9% | 常规业务 |
| 长+complex+HIGH | ≤60s | ≥15 | ≥99.5% | 深度分析 |

#### 7.2.3 思考模式专项验收

**思考阶段性能**：
- 思考时长 P99 不超过 P95 的 1.5 倍
- 思考时间占比稳定在 60%-85% 范围内
- 不同推理等级的思考时长符合预期

**输出阶段性能**：
- 输出 TPS 不低于传统模型
- 输出 TTFT 不受思考等级影响
- 输出质量满足业务需求

### 7.3 场景化验收要求（新增）

#### 7.3.1 场景覆盖率要求

**最小覆盖**：
- ✅ 至少 3 种推理等级
- ✅ 至少 3 种上下文长度
- ✅ 至少 3 种问题复杂度
- ✅ 至少 6 种场景组合

**推荐覆盖**：
- ✅ 所有 4 种推理等级
- ✅ 所有 3 种上下文长度
- ✅ 所有 3 种问题复杂度
- ✅ 重点 9 种场景组合

#### 7.3.2 场景性能一致性

**上下文长度影响**：
- 中等上下文 / 短上下文 ≤ 3.0
- 长上下文 / 短上下文 ≤ 6.0

**复杂度影响**：
- 复杂问题 / 简单问题 ≤ 5.0
- 中等问题 / 简单问题 ≤ 2.5

### 7.4 非流式接口

- 成功率 >= 99.9%
- P95 总响应时间按 Token 档位定义
- 峰值并发下无明显错误抬升

### 7.5 稳定性

- 连续运行 1 小时以上
- P95 性能波动 <= 20%
- 无明显资源泄漏
- **思考模式性能稳定**（新增）
- **场景间性能隔离有效**（新增）

### 7.6 容错

- 单模型故障时核心业务成功率 >= 99%
- fallback / 降级生效
- 不出现级联雪崩
- **思考模式超时能优雅降级**（新增）
- **长上下文场景不挤占短场景资源**（新增）

## 8. 工具落地建议

### 8.1 优先推荐 Locust

原因：

- 易于构造动态请求
- 易于统计 TTFT / TTLT
- Python 写法适合 AI 接口
- 更适合流式 SSE 场景

### 8.2 JMeter 适用场景

- HTTP 基础压测
- 非流式接口
- 团队已有 JMeter 使用经验
- 快速验证 QPS / 并发能力

### 8.3 建议组合

- 流式 chat：Locust
- 非流式 / 管理接口：JMeter 或 Locust 均可
- 长期建议统一为 `Locust + Prometheus + Grafana`

## 9. Locust 落地脚本骨架

```python
from locust import HttpUser, task, between
import time
import json


class AIGatewayUser(HttpUser):
    wait_time = between(1, 3)

    headers = {
        "Content-Type": "application/json",
        "Authorization": "Bearer your-token"
    }

    @task(7)
    def chat_stream(self):
        payload = {
            "model": "your-model",
            "stream": True,
            "messages": [
                {"role": "user", "content": "请总结一下AI网关性能测试的核心目标"}
            ],
            "temperature": 0.7
        }

        start = time.time()
        first_token_time = None
        token_count = 0

        with self.client.post(
            "/v1/chat/completions",
            data=json.dumps(payload),
            headers=self.headers,
            stream=True,
            catch_response=True,
            name="chat_stream"
        ) as resp:
            try:
                if resp.status_code != 200:
                    resp.failure(f"HTTP {resp.status_code}")
                    return

                for line in resp.iter_lines():
                    if not line:
                        continue

                    now = time.time()
                    if first_token_time is None:
                        first_token_time = now

                    token_count += 1

                end = time.time()

                ttft = (first_token_time - start) if first_token_time else None
                ttlt = end - start
                tps = token_count / (end - first_token_time) if first_token_time and end > first_token_time else 0

                resp.success()
                print(f"TTFT={ttft}, TTLT={ttlt}, TPS={tps}, tokens={token_count}")

            except Exception as e:
                resp.failure(str(e))

    @task(2)
    def chat_non_stream(self):
        payload = {
            "model": "your-model",
            "stream": False,
            "messages": [
                {"role": "user", "content": "请写一段100字左右的总结"}
            ]
        }

        with self.client.post(
            "/v1/chat/completions",
            data=json.dumps(payload),
            headers=self.headers,
            catch_response=True,
            name="chat_non_stream"
        ) as resp:
            if resp.status_code == 200:
                resp.success()
            else:
                resp.failure(f"HTTP {resp.status_code}")

    @task(1)
    def embeddings(self):
        payload = {
            "model": "your-embedding-model",
            "input": "AI网关压测示例文本"
        }

        with self.client.post(
            "/v1/embeddings",
            data=json.dumps(payload),
            headers=self.headers,
            catch_response=True,
            name="embeddings"
        ) as resp:
            if resp.status_code == 200:
                resp.success()
            else:
                resp.failure(f"HTTP {resp.status_code}")
```

## 10. 思考推理模型压测实施指南（新增）

### 10.1 思考推理模型压测流程

#### 10.1.1 测试前准备

**环境确认**：
```
✅ 网关支持思考推理API（如 /v1/responses）
✅ 配置推理等级参数（reasoning_effort）
```

### 10.3 思考推理模型验收检查清单

#### 10.3.1 指标验收检查

**思考阶段指标**：

- [ ] P99 思考时长不超过 P95 的 1.5 倍

- [ ] 输出 TTFT 不受思考等级影响
- [ ] 输出质量满足业务需求

**场景维度指标**：
- [ ] 上下文长度影响系数在合理范围
- [ ] 复杂度影响系数符合预期
- [ ] 不同场景间性能隔离有效

#### 10.3.2 性能基线建立

**等级性能基线**：
```
OFF等级：平均思考时长 0s，输出TTFT ≤ 1s
LOW等级：平均思考时长 3-5s，输出TTFT ≤ 1s
MEDIUM等级：平均思考时长 10-15s，输出TTFT ≤ 1s
HIGH等级：平均思考时长 30-45s，输出TTFT ≤ 2s
```

**场景性能基线**：
```
短+简单场景：平均思考时长 3-5s
中等+中等场景：平均思考时长 10-15s
长+复杂场景：平均思考时长 30-60s
```

### 10.4 思考推理模型压测常见问题

#### 问题1：思考时长异常增长

**现象**：P99 思考时长 > P95 × 2

**诊断步骤**：
1. 检查并发数是否合理
2. 分析不同场景的思考时长分布
3. 验证上游模型性能

**解决方案**：
- 降低并发数
- 优化长上下文场景
- 调整推理等级

#### 问题2：场景间性能干扰

**现象**：短场景被长场景拖慢

**诊断步骤**：
1. 分析场景间的资源竞争
2. 检查调度策略是否合理
3. 验证资源隔离效果

**解决方案**：
- 实施场景化资源隔离
- 设置优先级调度
- 限制长场景并发数

#### 问题3：资源耗尽

**现象**：思考模式占用过多资源

**诊断步骤**：
1. 分析思考模式的资源消耗
2. 检查资源分配策略
3. 验证资源限制配置

**解决方案**：
- 为思考模式设置独立资源池
- 实施资源配额限制
- 优化长场景处理策略

## 11. 场景化压测实施指南（新增）

### 11.1 场景化压测价值

**传统单一维度测试**：
```
只关注：推理等级（OFF/LOW/MEDIUM/HIGH）
问题：无法体现上下文和复杂度的影响

```

**场景化多维度测试**：
```
关注：推理等级 × 上下文长度 × 问题复杂度
优势：全面评估不同场景的性能特征
结果：提供精确的性能画像和优化建议
```

### 11.2 场景化压测实施策略

#### 11.2.1 渐进式场景覆盖

**阶段1：核心场景**（第1周）
```
必测场景（6个）：
1. 短+简单+OFF/LOW
2. 短+中等+MEDIUM
3. 中等+中等+MEDIUM
4. 长+complex+HIGH
5. 中等+complex+HIGH
```

```
扩展场景（12个）：
- 所有上下文长度 × 所有复杂度组合
- 每个组合配置2个推理等级

目标：全面评估场景影响，建立场景模型
```

**阶段3：优化验证**（第3周）
```
验证内容：
- 实施分级路由策略
- 验证场景隔离效果
- 对比优化前后性能

```

#### 11.2.2 场景化性能目标设定

**基于业务容忍度设定目标**：

| 业务类型 | 用户容忍度 | 性能目标 | 优先级 |
| --- | --- | --- | --- |
| 实时对话 | ≤3s | TTFT ≤ 1s | 速度 |
| 交互查询 | ≤10s | TTFT ≤ 5s | 速度 |
| 异步分析 | ≤60s | 质量优先 | 质量 |
| 批处理 | ≤300s | 质量优先 | 质量 |

**场景化性能目标映射**：

```
IF 实时对话 AND 短上下文 AND 简单问题:
    → OFF等级，TTFT ≤ 1s
ELSE IF 实时对话 AND 中等上下文 AND 中等问题:
    → LOW等级，TTFT ≤ 3s
ELSE IF 异步分析 AND 长上下文 AND 复杂问题:
    → HIGH等级，质量优先
```

### 11.4 场景化优化策略

#### 11.4.1 分级路由策略

**基于场景特征的智能路由**：

```python
def route_by_scenario(input_length, complexity, urgency, user_tier):
    # 时间敏感 + 短上下文 + 简单问题
    if urgency == "high" and input_length < 2000 and complexity == "simple":
        return {"reasoning": "off", "priority": "high"}

    # 质量优先 + 长上下文 + 复杂问题
    if urgency == "low" and input_length > 8000 and complexity == "complex":
        return {"reasoning": "high", "priority": "low"}

    # 默认：中等推理
    return {"reasoning": "medium", "priority": "normal"}
```

#### 11.4.2 资源隔离策略

**场景化资源池设计**：
```
资源池A（快速响应）：

- 超时限制：短超时（5s）

资源池B（平衡性能）：
- 适用场景：中等场景，MEDIUM等级
- 并发限制：标准并发（100%基准）
- 超时限制：标准超时（30s）

资源池C（质量优先）：
- 适用场景：长+复杂，HIGH等级
- 并发限制：低并发（30%基准）
- 超时限制：长超时（120s）
```
