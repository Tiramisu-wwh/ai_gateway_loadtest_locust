# AI 网关性能测试标准与指标说明文档

## 1. 文档目的

本规范用于定义 AI 网关在多模型接入场景下的性能指标体系、测试标准及验收口径，确保系统具备以下能力：

- 高可用性：稳定可靠
- 良好用户体验：低延迟、响应流畅
- 高承载能力：支撑业务增长
- 良好容错能力：模型异常不影响核心业务

## 2. 适用范围

本规范适用于 AI 网关接入的以下能力：

- 大模型推理（Chat / Completion / Function Calling）
- Embedding / Rerank
- 多模型路由（负载均衡 / fallback / 灰度）
- API 网关能力（鉴权、限流、转发、配额）

## 3. 指标体系设计原则

### 3.1 用户视角优先

指标应反映真实用户体验，而非仅关注系统内部指标。

- TTFT（首 Token 时间）优先于平均响应时间
- P95 / P99 优先于平均值

### 3.2 AI 场景特化

AI 接口与传统接口在性能衡量上存在差异：

| 传统接口 | AI 接口 |
| --- | --- |
| 响应时间 | TTFT + TTLT |
| QPS | 并发 + Token 吞吐 |
| 响应大小 | Token 数量 |

### 3.3 必须分档

禁止采用统一 SLA，必须按以下维度分档：

- Token 规模（小 / 中 / 大请求）
- 模型类型（轻量 / 大模型 / 多模态）
- 接口类型（流式 / 非流式）

## 4. 核心指标定义

### 4.1 可用性指标

#### 4.1.1 成功率（Success Rate）

定义：

```text
成功率 = 成功请求数 / 总请求数
```

指标说明：

表示请求最终成功返回有效结果的比例。

意义：

- 核心用户体验指标
- 直接影响业务可用性

建议标准：

- >= 99.9%

参考：

- SRE 服务可用性指标体系（SLI）

#### 4.1.2 错误率（Error Rate）

定义：

```text
错误率 = 错误请求数 / 总请求数
```

建议拆分：

- 网关错误（鉴权 / 参数 / 限流）
- 上游模型错误（429 / 5xx）
- 系统异常

建议标准：

- <= 0.1%

#### 4.1.3 超时率（Timeout Rate）

定义：

```text
超时请求数 / 总请求数
```

说明：

AI 场景中超时问题较常见，建议单独统计。

建议标准：

- <= 0.5%

### 4.2 核心性能指标（AI 重点）

#### 4.2.1 首 Token 时间（TTFT）

定义：

```text
请求发起 -> 返回第一个 Token 的时间
```

用户感知：

“多久开始回答”

意义：

- 决定交互流畅度
- 是 AI 体验的关键指标

建议标准：

- P95 <= 2s
- P99 <= 3s

参考：

- 云厂商 LLM Observability
- 行业 LLM Benchmark

#### 4.2.2 总响应时间（TTLT）

定义：

```text
请求发起 -> 返回最后一个 Token 的时间
```

说明：

用于衡量完整任务耗时。

注意：

必须按 Token 数量分档。

#### 4.2.3 Token 输出速率（Tokens/sec）

定义：

```text
生成 Token 数 / 生成时间
```

用户感知：

“回答是否流畅”

建议标准：

- >= 15-20 tokens/s（基础业务）
- 高性能目标：20-50 tokens/s

#### 4.2.4 延迟分位（P95 / P99）

定义：

- P95：95% 请求低于该时间
- P99：99% 请求低于该时间

意义：

- 反映长尾延迟
- 比平均值更真实

### 4.3 吞吐与容量指标

#### 4.3.1 QPS / RPS

定义：

```text
每秒请求数
```

说明：

用于衡量系统吞吐能力。

#### 4.3.2 并发数（Concurrency）

定义：

```text
同时处理的请求数量
```

意义：

- AI 场景中的核心指标
- 相比 QPS 更能反映真实压力

#### 4.3.3 Token 吞吐量

定义：

```text
每秒处理 Token 数（输入 + 输出）
```

意义：

- 更精确地衡量模型压力
- 相比 QPS 更真实

### 4.4 稳定性与资源指标

#### 4.4.1 性能稳定性

定义：

长时间运行中的性能波动情况。

测试方式：

- 持续压测 >= 1 小时

标准：

- 性能波动 <= 20%

#### 4.4.2 资源利用率

指标：

- CPU
- 内存
- 线程池
- 连接池

#### 4.4.3 限流触发率

定义：

```text
限流请求数 / 总请求数
```

#### 4.4.4 重试成功率

定义：

```text
重试成功请求 / 重试请求
```

意义：

体现网关容错能力，是核心价值指标之一。

## 5. 指标分档标准

### 5.1 按 Token 规模

| 档位 | 输入 Token |
| --- | --- |
| 小请求 | < 2k |
| 中请求 | 2k-8k |
| 大请求 | > 8k |

### 5.2 按接口类型

- 流式接口（Chat Streaming）
- 非流式接口
- Embedding
- Rerank
- 管理接口

### 5.3 按模型类型

- 小模型（轻量）
- 大模型（GPT 类）
- 多模态模型

## 6. 验收标准

### 6.1 核心推理接口

| 指标 | 标准 |
| --- | --- |
| 成功率 | >= 99.9% |
| 错误率 | <= 0.1% |
| 超时率 | <= 0.5% |
| P95 TTFT | <= 2s |
| P99 TTFT | <= 3s |
| Token 速率 | >= 15 tokens/s |
| 稳定性 | >= 1 小时无明显劣化 |

### 6.2 容量要求

- 最大稳定并发 >= 业务峰值 × 1.5~2

### 6.3 容错能力

- 单模型异常时，成功率 >= 99%
- 自动 fallback 生效
- 不影响核心业务

## 7. 测试场景要求

必须覆盖以下场景：

1. 基线测试（单用户）
2. 阶梯升压测试（找拐点）
3. 稳定性测试（>= 1 小时）
4. 峰值突刺测试（瞬时高并发）
5. 混合流量测试（多接口）
6. 异常测试（模型故障 / 超时）

## 8. 结论

AI 网关性能测试的核心目标不是单纯追求高 QPS，而是验证以下能力：

- 是否足够快：TTFT
- 是否足够流畅：Token 速率
- 是否具备承载能力：并发能力
- 是否具备容错能力：异常情况下的稳定性

## 9. 后续建设建议

- 建立标准压测模型（基于真实业务流量）
- 建立性能测试报告模板
- 建立 SLA 监控体系
- 建立容量预测与扩容机制
