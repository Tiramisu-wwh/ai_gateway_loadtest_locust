# AI网关整体性能测试标准

## 1. 文档目的

本规范用于定义 AI 网关在多模型接入场景下的性能指标体系、测试标准及验收口径，确保系统具备以下能力：

- 高可用性：稳定可靠
- 良好用户体验：低延迟、响应流畅
- 高承载能力：支撑业务增长
- 良好容错能力：模型异常不影响核心业务

## 2. 适用范围

本规范适用于 AI 网关接入的以下能力：

- 大模型推理（Chat / Completion / Function Calling）
- Embedding / Rerank
- 多模型路由（负载均衡 / fallback / 灰度）
- API 网关能力（鉴权、限流、转发、配额）

## 2.1 测试对象说明

本规范定义的测试方法是**端到端性能测试**，测试路径为：

```
测试工具（Locust） -> AI网关 -> 后端模型服务
                      测试整体性能
```

**核心特点**：
- 使用真实后端模型服务
- 测试目标是用户体验层面的性能
- 关注整体系统的承载能力

**适用场景**：
- 日常性能监控与回归测试
- 生产容量规划
- 性能基线建立与对比
- 思考推理模型性能验证

**注意**：如需测试网关自身的并发处理能力（不依赖后端模型），请参考《AI网关独立并发性能测试方案.md》。

## 3. 指标体系设计原则

### 3.1 用户视角优先

指标应反映真实用户体验，而非仅关注系统内部指标。

- TTFT（首 Token 时间）优先于平均响应时间
- P95 / P99 优先于平均值

### 3.2 AI 场景特化

AI 接口与传统接口在性能衡量上存在差异：

| 传统接口 | AI 接口 |
| --- | --- |
| 响应时间 | TTFT + TTLT |
| QPS | 并发 + Token 吞吐 |
| 响应大小 | Token 数量 |

### 3.3 必须分档

禁止采用统一 SLA，必须按以下维度分档：

- Token 规模（小 / 中 / 大请求）
- 模型类型（轻量 / 大模型 / 多模态）
- 接口类型（流式 / 非流式）

## 4. 核心指标定义

### 4.1 可用性指标

#### 4.1.1 成功率（Success Rate）

定义：

```text
成功率 = 成功请求数 / 总请求数
```

指标说明：

表示请求最终成功返回有效结果的比例。

意义：

- 核心用户体验指标
- 直接影响业务可用性

建议标准：

- >= 99.9%

参考：

- SRE 服务可用性指标体系（SLI）

#### 4.1.2 错误率（Error Rate）

定义：

```text
错误率 = 错误请求数 / 总请求数
```

建议拆分：

- 网关错误（鉴权 / 参数 / 限流）
- 上游模型错误（429 / 5xx）
- 系统异常

建议标准：

- <= 0.1%

#### 4.1.3 超时率（Timeout Rate）

定义：

```text
超时请求数 / 总请求数
```

说明：

AI 场景中超时问题较常见，建议单独统计。

建议标准：

- <= 0.5%

### 4.2 核心性能指标（AI 重点）

#### 4.2.1 首 Token 时间（TTFT）

定义：

```text
请求发起 -> 返回第一个 Token 的时间
```

用户感知：

“多久开始回答”

意义：

- 决定交互流畅度
- 是 AI 体验的关键指标

建议标准：

- P95 <= 2s
- P99 <= 3s

参考：

- 云厂商 LLM Observability
- 行业 LLM Benchmark

#### 4.2.2 总响应时间（TTLT）

定义：

```text
请求发起 -> 返回最后一个 Token 的时间
```

说明：

用于衡量完整任务耗时。

注意：

必须按 Token 数量分档。

#### 4.2.3 Token 输出速率（Tokens/sec）

定义：

```text
生成 Token 数 / 生成时间
```

用户感知：

“回答是否流畅”

建议标准：

- >= 15-20 tokens/s（基础业务）
- 高性能目标：20-50 tokens/s

#### 4.2.4 延迟分位（P95 / P99）

定义：

- P95：95% 请求低于该时间
- P99：99% 请求低于该时间

意义：

- 反映长尾延迟
- 比平均值更真实

### 4.3 吞吐与容量指标

#### 4.3.1 QPS / RPS

定义：

```text
每秒请求数
```

说明：

用于衡量系统吞吐能力。

#### 4.3.2 并发数（Concurrency）

定义：

```text
同时处理的请求数量
```

意义：

- AI 场景中的核心指标
- 相比 QPS 更能反映真实压力

#### 4.3.3 Token 吞吐量

定义：

```text
每秒处理 Token 数（输入 + 输出）
```

意义：

- 更精确地衡量模型压力
- 相比 QPS 更真实

### 4.4 思考推理模型专用指标

#### 4.4.1 思考推理模型特点

思考推理模型（如 OpenAI o1 系列、Claude 推理增强模型等）在输出最终答案前会先进行内部推理思考，这改变了传统 AI 模型的性能特征：

| 传统模型 | 思考推理模型 |
| --- | --- |
| 请求 → 直接生成 | 请求 → 思考 → 生成 |
| TTFT = 首个生成 Token | TTFT = 思考结束 + 首个生成 Token |
| 单一延迟指标 | 区分思考延迟 + 生成延迟 |

#### 4.4.2 思考阶段指标

**思考时长（Thinking Duration）**

定义：

```text
从请求发起到思考结束、开始输出答案的时间
```

用户感知：

"模型在想多久"

意义：

- 反映模型推理复杂度
- 影响用户等待体验
- 与推理等级（effort）正相关

建议标准：

- P95 <= 30s（中等推理强度）
- P99 <= 60s
- 平均值 <= 20s

**思考 Token 数（Thinking Tokens）**

定义：

```text
思考过程中产生的 Token 数量（如果 API 返回）
```

说明：

- 部分 API 不暴露此指标
- 可用于估算思考复杂度

#### 4.4.3 输出阶段指标

**输出 TTFT（Output TTFT）**

定义：

```text
从思考结束到第一个输出 Token 的时间
```

说明：

- 思考模型专用指标
- 反映答案生成的起始速度

建议标准：

- P95 <= 1s
- P99 <= 2s

**输出 TPS（Output Tokens Per Second）**

定义：

```text
输出阶段 Token 生成速率
```

说明：

- 区别于传统模型的全程 TPS
- 仅计算输出阶段的速率

建议标准：

- 平均 >= 20 tokens/s
- P95 >= 15 tokens/s

#### 4.4.4 组合指标

**端到端延迟（End-to-End Latency）**

定义：

```text
从请求开始到最后一个 Token 的总时间
```

组成：

```text
端到端延迟 = 思考时长 + 输出时长
```

**思考时间占比（Thinking Ratio）**

定义：

```text
思考时间占比 = 思考时长 / 端到端延迟
```

意义：

- 反映计算资源分配
- 典型值：60%-85%
- 过低可能思考不足，过高可能输出很少

#### 4.4.5 不同推理等级的性能预期

| 推理等级 | 预期思考时长 | 预期输出时长 | 适用场景 |
| --- | --- | --- | --- |
| OFF | 0s | 正常 | 简单问答、快速响应 |
| LOW | 2-5s | 正常 | 简单推理、常规任务 |
| MEDIUM | 5-15s | 正常 | 复杂推理、深度分析 |
| HIGH | 15-60s | 正常 | 高度复杂推理、专业问题 |

#### 4.4.6 思考模式测试要求

**必须区分测试场景**：

1. 不同推理等级独立测试
2. 对比基准：思考 vs 非思考
3. 重点关注思考阶段的性能稳定性

**指标采集要求**：

1. 必须记录推理等级（reasoning_effort）
2. 必须区分思考和输出阶段的时长
3. 建议记录思考内容长度（如果 API 支持）

**性能目标设定原则**：

1. 思考时长目标与推理等级匹配
2. 输出阶段性能不应低于传统模型
3. 端到端延迟需符合业务场景容忍度

### 4.5 稳定性与资源指标

#### 4.4.1 性能稳定性

定义：

长时间运行中的性能波动情况。

测试方式：

- 持续压测 >= 1 小时

标准：

- 性能波动 <= 20%

#### 4.4.2 资源利用率

指标：

- CPU
- 内存
- 线程池
- 连接池

#### 4.4.3 限流触发率

定义：

```text
限流请求数 / 总请求数
```

#### 4.4.4 重试成功率

定义：

```text
重试成功请求 / 重试请求
```

意义：

体现网关容错能力，是核心价值指标之一。

## 5. 指标分档标准

### 5.1 按 Token 规模分档

| 档位 | 输入 Token | 输出 Token | 测试重点 |
| --- | --- | --- | --- |
| 小请求 | < 2k | < 500 | 快速响应、低延迟 |
| 中请求 | 2k-8k | 500-1500 | 平衡性能与质量 |
| 大请求 | > 8k | > 1500 | 吞吐能力、资源利用 |

### 5.2 按接口类型分档

- 流式接口（Chat Streaming）- 重点关注 TTFT
- 非流式接口 - 重点关注总响应时间
- Embedding - 重点关注吞吐量
- Rerank - 重点关注准确性
- 管理接口 - 重点关注响应时间

### 5.3 按模型类型分档

- 小模型（轻量）- 侧重并发能力
- 大模型（GPT 类）- 侧重延迟和吞吐
- 推理型模型 - 侧重思考质量和时间
- 多模态模型 - 侧重各模态协调性能

### 5.4 按思考推理等级分档

| 推理等级 | 思考时长范围 | 输出性能要求 | 适用场景 |
| --- | --- | --- | --- |
| OFF | 0s | TTFT ≤ 1s | 简单问答、快速响应 |
| LOW | 2-5s | TPS ≥ 20 | 简单推理、常规任务 |
| MEDIUM | 5-15s | TPS ≥ 20 | 复杂推理、深度分析 |
| HIGH | 15-60s | TPS ≥ 15 | 高度复杂推理、专业问题 |

### 5.5 按上下文场景分档（新增）

#### 5.5.1 上下文长度分档

| 档位 | 输入 Token 范围 | 性能特征 | 测试重点 |
| --- | --- | --- | --- |
| 短上下文 | < 2k | 快速响应、低延迟 | TTFT、响应速度 |
| 中等上下文 | 2k-8k | 平衡性能和准确性 | 综合性能、稳定性 |
| 长上下文 | > 8k | 深度理解、高准确性 | 吞吐能力、资源利用 |

#### 5.5.2 问题复杂度分档

| 复杂度 | 问题特征 | 推理需求 | 性能预期 |
| --- | --- | --- | --- |
| 简单 | 事实查询、简单计算 | 无需/轻度思考 | 快速响应（≤3s） |
| 中等 | 推理分析、对比说明 | 中度思考 | 平衡响应（5-15s） |
| 复杂 | 多步推理、综合分析 | 深度思考 | 质量优先（15-60s） |

#### 5.5.3 场景组合矩阵

实际测试中需要考虑上下文长度和复杂度的组合影响：

| 场景组合 | 输入特征 | 复杂度特征 | 推荐推理等级 | 性能目标重点 |
| --- | --- | --- | --- | --- |
| 短+简单 | < 2k tokens | 事实查询 | OFF/LOW | 快速响应 TTFT ≤ 1s |
| 短+中等 | < 2k tokens | 简单推理 | LOW/MEDIUM | 响应速度 TTFT ≤ 3s |
| 中等+中等 | 2k-8k tokens | 推理分析 | MEDIUM | 综合性能 平衡质量和速度 |
| 长+复杂 | > 8k tokens | 深度推理 | HIGH | 质量优先 确保准确性 |

### 5.6 场景维度对性能的影响

#### 5.6.1 上下文长度的影响分析

**影响规律**：
```
思考时长 ≈ 基础时间 + (输入Token数 × 系数) × 复杂度系数
```

**实测数据参考**：
- 短上下文（500 tokens）：思考时长 3-5s
- 中等上下文（3000 tokens）：思考时长 8-15s
- 长上下文（10000+ tokens）：思考时长 20-60s

**性能目标设定**：
```
短上下文场景：P95 思考时长 ≤ 10s
中等上下文场景：P95 思考时长 ≤ 30s
长上下文场景：P95 思考时长 ≤ 60s
```

#### 5.6.2 问题复杂度的影响分析

**影响规律**：
```
简单问题：思考时长 ≈ 基准 × 1.0
中等问题：思考时长 ≈ 基准 × 2.0
复杂问题：思考时长 ≈ 基准 × 4.0
```

**分级路由建议**：
- 简单问题：使用 OFF 或 LOW 等级，优先速度
- 中等问题：使用 MEDIUM 等级，平衡质量和速度
- 复杂问题：使用 HIGH 等级，优先质量

## 6. 场景化测试标准

### 6.1 思考推理模型测试标准

#### 6.1.1 测试场景设计

**基础测试矩阵**：

| 测试类型 | 推理等级 | 上下文长度 | 问题复杂度 | 预期目标 |
| --- | --- | --- | --- | --- |
| 思考等级基准 | OFF/LOW/MEDIUM/HIGH | 固定中等 | 固定中等 | 对比不同等级性能 |
| 场景影响分析 | MEDIUM | 短/中/长 | 简单/中/复杂 | 量化场景影响 |
| 混合负载测试 | 混合 | 混合 | 混合 | 验证真实负载 |

#### 6.1.2 性能目标分档

**按推理等级和场景分档的性能目标**：

| 推理等级 | 场景类型 | P95 思考时长 | P99 思考时长 | 输出 TPS | 适用性说明 |
| --- | --- | --- | --- | --- | --- |
| OFF | 所有场景 | N/A | N/A | ≥20 | 快速响应场景 |
| LOW | 短+简单 | ≤5s | ≤8s | ≥20 | 常规快速问答 |
| LOW | 中等+中等 | ≤8s | ≤12s | ≥20 | 简单推理任务 |
| MEDIUM | 短+中等 | ≤10s | ≤15s | ≥20 | 中等推理场景 |
| MEDIUM | 中等+中等 | ≤20s | ≤30s | ≥20 | 复杂推理任务 |
| MEDIUM | 长+复杂 | ≤45s | ≤60s | ≥15 | 深度分析场景 |
| HIGH | 中等+复杂 | ≤40s | ≤60s | ≥15 | 专业级分析 |
| HIGH | 长+复杂 | ≤60s | ≤90s | ≥10 | 顶级复杂推理 |

### 6.2 场景维度测试要求

#### 6.2.1 必须覆盖的场景组合

**最小测试集**（6个核心场景）：
1. 短上下文 + 简单问题 + OFF/LOW 等级
2. 短上下文 + 中等问题 + MEDIUM 等级
3. 中等上下文 + 中等问题 + MEDIUM 等级
4. 中等上下文 + 复杂问题 + HIGH 等级
5. 长上下文 + 中等问题 + MEDIUM 等级
6. 长上下文 + 复杂问题 + HIGH 等级

**完整测试集**（9个场景组合 × 4个推理等级 = 36种组合）

#### 6.2.2 场景化指标采集要求

**必须采集的场景元数据**：
1. 上下文长度标识（short/medium/long）
2. 问题复杂度标识（simple/moderate/complex）
3. 实际输入 Token 数
4. 推理等级标识
5. 场景组合标识

**场景化性能分析**：
1. 按上下文长度分组统计
2. 按复杂度分组统计
3. 计算场景影响系数
4. 识别异常场景模式

## 7. 验收标准

### 6.1 核心推理接口

| 指标 | 标准 |
| --- | --- |
| 成功率 | >= 99.9% |
| 错误率 | <= 0.1% |
| 超时率 | <= 0.5% |
| P95 TTFT | <= 2s |
| P99 TTFT | <= 3s |
| Token 速率 | >= 15 tokens/s |
| 稳定性 | >= 1 小时无明显劣化 |

### 6.2 容量要求

- 最大稳定并发 >= 业务峰值 × 1.5~2

### 6.3 容错能力

- 单模型异常时，成功率 >= 99%
- 自动 fallback 生效
- 不影响核心业务

## 7. 测试场景要求

必须覆盖以下场景：

1. 基线测试（单用户）
2. 阶梯升压测试（找拐点）
3. 稳定性测试（>= 1 小时）
4. 峰值突刺测试（瞬时高并发）
5. 混合流量测试（多接口）
6. 异常测试（模型故障 / 超时）

## 8. 结论

AI 网关性能测试的核心目标不是单纯追求高 QPS，而是验证以下能力：

- 是否足够快：TTFT
- 是否足够流畅：Token 速率
- 是否具备承载能力：并发能力
- 是否具备容错能力：异常情况下的稳定性

## 9. 后续建设建议

- 建立标准压测模型（基于真实业务流量）
- 建立性能测试报告模板
- 建立 SLA 监控体系
- 建立容量预测与扩容机制
